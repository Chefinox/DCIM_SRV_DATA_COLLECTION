## P0-1 — Pulihkan Elasticsearch (Avro deserialization)

**Akar:** `telegraf-consumer.service` membaca topik Avro dengan `data_format=json` → `invalid character '\x00'`.

**Opsi A (disarankan):** Ganti consumer ES dari Telegraf ke **consumer Python Avro khusus** (pola yang sama dengan `dcim-sql-consumer`), yang men-deserialize Avro via Schema Registry lalu bulk-index ke ES.
- Reuse modul deserialisasi Avro yang sudah dipakai sql-consumer.
- Tulis ke `dcim-metrics-unified-%Y.%m.%d` dengan timestamp yang benar (perbaiki bug indeks "2227").
- Buat sebagai `dcim-es-consumer.service`.

**Opsi B (cepat, sementara):** Arahkan input `telegraf-consumer` ke topik **JSON** (`dcim.raw.*` atau topik JSON terpisah), bukan Avro. Kekurangan: kehilangan field hasil enrichment.

**Bersihkan:** hapus indeks kosong `dcim-metrics-unified-2227.*`.

**Verifikasi:** dokumen baru muncul di ES dengan `@timestamp` hari ini; tidak ada error `\x00` di journal.

## P0-2 — Putuskan nasib Flink

Karena implementasi yang diklaim **tidak ada**, dua jalur:

**Opsi A — Realisasikan real-time alerting (sejalan dcim-wiki).**
1. Buat `src/skills/telemetry/alerter/flink_alerter.py` PyFlink **sungguhan**, di-commit ke git.
2. Consume **`dcim.enriched.events` (Avro)** — bukan raw — agar punya konteks enrichment; sertakan deserializer Avro/Schema Registry.
3. Output ke topik yang **punya consumer nyata** (mis. reuse jalur DLQ/alert yang dikonsumsi telegram-alerter), bukan `dcim.alerts.p1` dead-end.
4. `flink/docker-compose.yml`: tambah section `networks` (join ke network Kafka) **secara deklaratif**, aktifkan **checkpointing** + `restart: unless-stopped`, dan mekanisme submit job otomatis.
5. Verifikasi job RUNNING setelah `docker compose down/up` (uji durabilitas).

**Opsi B — Putuskan resmi, pakai alerting yang ada.** Jika Flink belum prioritas: hapus `flink/` dari repo, koreksi `task.md`/walkthrough agent (tandai "NOT DONE"), dan andalkan threshold alerter (setelah P1-2). Catat di dokumen arsitektur sebagai keputusan sadar.

> Rekomendasi: **Opsi B sekarang** (hentikan klaim palsu, stabilkan), jadwalkan **Opsi A** sebagai pekerjaan terencana — bukan ephemeral.

## P1-1 — Hentikan banjir DLQ delivery-failure

**Akar:** event `metric_value: null` / `general_metric` (NAS SNMP, network interfaces) ditolak saat delivery dan dibuang ke DLQ ~30/dtk.

**Langkah:**
1. **Filter di hulu:** di normalizer/consumer, jangan emit/route event dengan `metric_value IS NULL` & `metric_name='general_metric'` (atau map ke metrik yang valid).
2. **Perbaiki SNMP mapping NAS/interfaces** agar menghasilkan metric bernilai, bukan `general_metric` kosong.
3. Tetapkan **retention** wajar pada topik DLQ (mis. `retention.ms` 7 hari) agar tidak tumbuh tanpa batas.
4. Drain/kompaksi 3,98 jt pesan lama setelah penyebab dihentikan.

**Verifikasi:** laju delivery-failure ≈ 0/dtk setelah fix.
