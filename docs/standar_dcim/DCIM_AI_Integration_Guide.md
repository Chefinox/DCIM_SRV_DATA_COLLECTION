# DCIM: AI Integration & Data Ingestion Guide

This document is designed to provide all technical details necessary for an external AI to interact with the DCIM Source of Truth (SoT) environment for **Normalization**, **Enrichment**, and **ETL** processes.

## 1. System Architecture
The DCIM platform aggregates data from two primary sources into a centralized PostgreSQL Data Warehouse.

- **NetBox**: Primary source for logical infrastructure (Sites, VLANs, VRFs, Logical IPs).
- **Ralph**: Primary source for physical asset lifecycle (Hardware SN, Models, Firmware, BIOS).
- **Source of Truth (SoT)**: PostgreSQL 15 database where normalized data is stored.

## 3. Data Schema: `unified_assets` Table
The central table used for normalization is `unified_assets` in the `dcim_sot` database.

| Column | Type | Description |
| :--- | :--- | :--- |
| `id` | `SERIAL (PK)` | Internal database ID. |
| `source_system` | `VARCHAR` | "NetBox" or "Ralph". |
| `asset_id` | `VARCHAR` | The ID of the asset in the original system. |
| `hostname` | `VARCHAR` | Device hostname or label. |
| `serial_number` | `VARCHAR` | Hardware Serial Number. |
| `category` | `VARCHAR` | Device Model or Category. |
| `site` | `VARCHAR` | Location or Site name. |
| `raw_payload` | `JSONB` | **Crucial:** Full JSON response from the original API. |
| `last_synced` | `TIMESTAMP` | Time of the last ETL run. |

## 4. Normalization Logic
The ETL pipeline (`unified_sot_etl.py`) maps fields as follows:

### NetBox Mapping
- `asset_id` ← `id`
- `hostname` ← `name`
- `serial_number` ← `serial`
- `category` ← `device_type.model`
- `site` ← `site.name`

### Ralph Mapping
- `asset_id` ← `id`
- `hostname` ← `hostname`
- `serial_number` ← `sn`
- `category` ← `model.name`
- `site` ← "Local Instance" (Static)

## 5. Enrichment Guidelines for AI
The `raw_payload` (JSONB) column contains extended attributes that can be used for enrichment:

### Example: Enriching with Management IP
For a NetBox device, the management IP is often nested:
`raw_payload -> 'primary_ip' -> 'address'`

### Example: Enriching with Firmware (Ralph)
Ralph often stores firmware in custom fields or attributes:
`raw_payload -> 'custom_fields' -> 'firmware_version'`

### Target Enrichment Schema
When processing telemetry, follow the [telemetry_schema.json](file:///d:/antigravity/DCIM/telemetry_schema.json) which requires:
- `asset_id`
- `serial_number`
- `category` (Enum: Server, NMS, Security, EPMS, BMS)
